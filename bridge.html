22.1 环境注册与轮询

bridgeApi.ts（540 行）封装了 Bridge 的 8 个 REST 端点，所有请求都通过 withOAuthRetry 实现 401 自动重试。

sequenceDiagram participant CLI as Claude Code CLI participant API as Bridge API
bridgeApi.ts participant Server as Anthropic Server CLI->>API: registerBridgeEnvironment API->>Server: POST /v1/environments/bridge Server-->>API: environment_id + secret loop Poll 循环 CLI->>API: pollForWork(env_id, secret) API->>Server: GET .../work/poll alt 有工作 Server-->>API: WorkResponse(id, data, token) CLI->>API: acknowledgeWork(work_id) else at-capacity CLI->>API: heartbeatWork(work_id) Server-->>API: lease_extended + state end end CLI->>API: stopWork / deregisterEnvironment
端点方法用途错误处理
/v1/environments/bridgePOST注册 Bridge 环境（发送 machine_name, dir, branch, worker_type）支持 reuseEnvironmentId 幂等重注册
.../work/pollGET轮询工作项（每 N 次空轮询才记日志，降噪）404 → 环境已过期
.../work/{id}/ackPOST确认工作项已被处理—
.../work/{id}/stopPOST释放工作项（force=true 强制终止）OAuth 重试
.../work/{id}/heartbeatPOST租约续期，返回 lease_extended + statelease 过期 → 重连

.../bridge/reconnectPOST重连现有会话到环境尝试 session_*/cse_* 两种 ID 格式

bridgeApi 安全设计

  • URL 路径注入防护：SAFE_ID_PATTERN = /^[a-zA-Z0-9_-]+$/ 校验所有 Bridge ID
  • OAuth 401 自动重试：withOAuthRetry 模式与 withRetry.ts 一致——刷新 token 后重试一次
  • 分级错误处理：401 → BridgeFatalError(auth)；403 → isExpiredErrorType 检查；404 → 环境过期；410 → 注销；429 → 限流
🔒 bridgeApi 安全设计
// URL 路径注入防护
const SAFE_ID_PATTERN = /^[a-zA-Z0-9_-]+$/
function validateBridgeId(id: string, label: string): string {
  if (!id || !SAFE_ID_PATTERN.test(id)) {
    throw new Error(`Invalid ${label}: contains unsafe characters`)
  }
  return id
}

// OAuth 401 自动重试（与 withRetry.ts 相同模式）
async function withOAuthRetry(fn, context) {
  const response = await fn(accessToken)
  if (response.status !== 401) return response
  // 尝试刷新 token
  const refreshed = await deps.onAuth401(accessToken)
  if (refreshed) {
    return await fn(resolveAuth()) // 重试一次
  }
  return response // 交给 handleErrorStatus 抛 BridgeFatalError
}

// 分级错误处理
// 401 → BridgeFatalError(auth)
// 403 → 检查 isExpiredErrorType + isSuppressible403
// 404 → BridgeFatalError(not found)
// 410 → 环境过期
// 429 → 限流
// 其他 → 通用 Error

22.2 REPL Bridge 核心

replBridge.ts（2,407 行）是单 CLI 会话的 Bridge 核心。它管理环境注册 → 会话创建 → Poll 循环 → Ingress WebSocket → 优雅关闭的完整生命周期。

3 层重连策略详解

stateDiagram-v2 [*] --> Ready: initBridgeCore Ready --> Connected: pollForWork 收到 work Connected --> Connected: heartbeatWork 续期 Connected --> Reconnecting: 环境丢失 / WS 异常关闭 Reconnecting --> Connected: tryReconnectInPlace 成功 Reconnecting --> Connected: fresh session 回退成功 Reconnecting --> Failed: 超过 MAX_ENVIRONMENT_RECREATIONS Connected --> [*]: teardown Failed --> [*]
机制实现设计意图
3 层重连doReconnect① tryReconnectInPlace（同 env 重连，URL 不变）→ ② fresh session（新会话）→ ③ give up（超 3 次）
BoundedUUIDSet环形缓冲区（容量 2000）Echo 过滤（recentPostedUUIDs）+ 重投去重（recentInboundUUIDs），O(1) add/has，O(capacity) 内存
FlushGateflushGate.ts初始消息刷送期间排队新消息，防止乱序到达服务器
SSE 序列号记忆lastTransportSequenceNum跨传输保持，重连时告知服务器已收到的最后事件 → 避免全量重播
Perpetual 模式bridgePointer.tscrash-recovery 指针（sessionId + environmentId），进程崩溃后可恢复
CapacityWakecapacityWake.ts会话完成 / WS 断开 → 信号唤醒 poll 休眠，立即切换到快速轮询
🔄 3 层重连策略 — doReconnect 深度解析
doReconnect() 流程：

1. 递增 environmentRecreations（超过 3 次 → 放弃）
2. 递增 v2Generation（使过时的 v2 传输握手失效）
3. 关闭旧 transport → 保存 SSE 序列号
4. wakePollLoop（唤醒 at-capacity 休眠）
5. flushGate.drop（重置保序门控）
6. 释放 currentWorkId（best-effort stopWork）

策略 1：同环境原地重连
  ├── re-register with reuseEnvironmentId
  ├── 如果 got == requested → tryReconnectInPlace
  │   ├── reconnectSession(session_* ID)
  │   └── reconnectSession(cse_* ID)  ← CCR v2 兼容
  └── 成功 → URL 不变，previouslyFlushedUUIDs 保留

策略 2：新会话回退
  ├── archiveSession(旧会话)
  ├── getCurrentTitle（可能被 /rename 修改）
  ├── createSession(新会话)
  └── 重置 SSE 序列号（新会话从 1 开始）

并发安全：
  ├── reconnectPromise 保证单次执行
  ├── 每个 await 点后检查 pollController.signal.aborted
  └── v2Generation 检测过时的传输握手

// 服务器控制命令处理（10-14s 超时 — 必须及时响应）
function handleServerControlRequest(request, handlers) {
  // outbound-only 模式（仅输出，不接受输入）
  // → initialize 必须成功（否则服务器断开连接）
  // → 其他命令返回 error 控制响应
  if (outboundOnly && request.request.subtype !== 'initialize') {
    response = { type: 'control_response', response: {
      subtype: 'error', request_id: ...,
      error: 'This session is outbound-only...'
    }}
  }

  // 正常模式 5 种命令：
  // initialize   → 返回 commands=[], models=[], pid
  // set_model    → 调用 onSetModel
  // interrupt    → 调用 onInterrupt
  // set_permission_mode → 策略校验后调用 onSetPermissionMode
  // set_max_thinking_tokens → 调用 onSetMaxThinkingTokens
  // 未知 subtype → error 响应（防止服务器挂起）
}

🔑 一句话总结：bridge/ 是 Claude Code 的 远程控制通信枢纽。bridgeApi 封装 8 个 REST 端点 + OAuth 401 自动重试 + 路径注入防护；replBridge 实现 3 层重连策略（原地重连 → 新会话 → 放弃）+ BoundedUUIDSet 环形缓冲去重 + FlushGate 保序 + SSE 序列号跨传输记忆；bridgeMessaging 用纯函数实现消息路由 + 5 种控制命令处理（10-14s 超时约束）；传输层支持 v1/v2 双协议（WS/SSE）+ JWT 预刷新 + 设备信任 + crash-recovery 指针。这 31 个文件共同构建了一个高可靠的实时双向通信系统。
