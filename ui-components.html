17.1 React Compiler 编译优化

Claude Code 全量使用了 React Compiler（react/compiler-runtime），所有组件在编译阶段被自动 memo 化。这不是实验性特性——它是性能架构的基石。

▶ _c() 编译器运行时的工作原理 App.tsx:1-55
// 编译前（原始 TSX）：
export function App({ getFpsMetrics, stats, initialState, children }) {

      </StatsProvider>

// 编译后（React Compiler 输出）：
export function App(t0) {
  const $ = _c(9);  // ← 分配 9 个缓存槽位
  const { getFpsMetrics, stats, initialState, children } = t0;
  let t1;
  if ($[0] !== children || $[1] !== initialState) {
    // 只有 children 或 initialState 变了才重新创建子树
    t1 = <AppStateProvider ...>{children}</AppStateProvider>;
    $[0] = children; $[1] = initialState; $[2] = t1;
  } else {
    t1 = $[2];  // ← 复用上次的 JSX 元素
  }

}

// 开发者不需要手写 useMemo/useCallback——编译器自动处理

▶ UUID 锚定切片算法 Messages.tsx:278-340
// 问题：非虚拟滚动模式下，Ink 为每条消息挂载完整 Fiber 树（~250KB RSS/条）
// 2000 条消息 → ~500MB Fiber + 3000 行 screen buffer
// → GC 死亡螺旋（观测到 59GB RSS、14K mmap/munmap/sec）

const MAX_MESSAGES_WITHOUT_VIRTUALIZATION = 200; // 安全上限
const MESSAGE_CAP_STEP = 50;  // 量子步进（不是逐条推进）

// 为什么不用 slice(-200)？
// count-based 切片在每次追加时丢弃前端一条消息，
// 导致终端 scrollback 内容整体偏移 → 每轮完整 terminal reset（CC-941）
// 50 条步进 quantize 也有问题——compaction/collapse 会改变 .length

// 解决：UUID 锚定
type SliceAnchor = { uuid: string; idx: number } | null;

function computeSliceStart(collapsed, anchorRef, cap, step) {
  const anchor = anchorRef.current;
  // 找到 anchor UUID 在当前数组中的位置
  const anchorIdx = anchor ? collapsed.findIndex(m => m.uuid === anchor.uuid) : -1;
  // UUID 找到 → 用它；UUID 丢失 → 降级到 stored index（clamped）
  let start = anchorIdx >= 0 ? anchorIdx
    : anchor ? Math.min(anchor.idx, Math.max(0, collapsed.length - cap))
    : 0;
  // 超出 cap+step 才推进窗口
  if (collapsed.length - start > cap + step) {
    start = collapsed.length - cap;
  }
  // 刷新锚点
  anchorRef.current = { uuid: collapsed[start].uuid, idx: start };
  return start;
}

            p.every((item, i) => item.contentBlock === n[i]?.contentBlock))

      if (key === 'tools') {
        if (p.every((tool, i) => tool.name === n[i]?.name))
          continue;
      }
      return false; // 真的变了，重渲染
    }
  }
  return true; // 跳过重渲染

31.3 Resolver 解析引擎 (resolver.ts)

Resolver 是键绑定系统的 运行时决策核心，支持单键和 Chord 两种解析模式：

关键设计决策

  • "Last Wins" 规则：默认绑定先加载，用户 keybindings.json 后加载——同键同上下文的后定义覆盖先定义
  • null 动作解绑：用户可以将动作设为 null 来禁用某个快捷键——chordWinners Map 追踪是否有非 null 的更长 Chord，null 覆盖不会让 chord-wait 误触发
  • Alt/Meta 合并：传统终端无法区分 Alt 和 Meta，keystrokesEqual() 将两者视为同一个逻辑修饰键；Super (cmd/win) 保持独立——仅通过 Kitty 键盘协议到达
  • Escape 修饰键修正：Ink 遗留行为将 escape 按键设置 key.meta=true，buildKeystroke() 特殊处理——对 escape 键本身忽略 meta 标志，否则 Chord 匹配会失败
