🧠 上下文管理 16

上下文管理是 Claude Code 架构中最关键但最容易被误解的模块。它解决一个根本问题：LLM 是无状态的，每次 API 调用都从零开始——于是必须在每次请求中精心构建 System Prompt，把"AI 应该知道的一切"塞进有限的 Token 窗口。

🔑 本章方法论：以下所有架构声明均源自 Harness 穷举分析——先扫描 1884 个 TS 文件的全局模式（buildTool/sideQuery/memoize/feature flag），再逐文件深入。每个声明标注源码位置。

16.1 搜索策略：为什么用 Grep 而不用 RAG

这是整个架构中最重要的设计决策之一，但从外部文档中完全看不出来：Claude Code 的代码搜索完全基于 ripgrep（正则匹配），不使用任何向量数据库、Embedding 模型或 RAG 管线。

⚠️ 穷举证据：在 1884 个 TypeScript 源文件中搜索 embedding、vector.*db、cosine.*simil、faiss、chroma、pinecone，结果为 0 条。这不是疏忽——这是深思熟虑的架构选择。
维度Claude Code 的选择典型 RAG 方案
搜索引擎ripgrep（正则匹配）向量数据库 + embedding
索引无索引，每次实时扫描文件系统需要预先构建向量索引
结果确定性的，精确匹配语义相似度，可能有误召回
基础设施零依赖（rg 编译进二进制）需运行 Chroma/Pinecone 等
延迟20K 文件 < 100ms索引构建可达数分钟
一致性永远反映文件系统当前状态有索引过期问题
成本完全免费（本地 CPU）每个文件需 API 调用生成向量

ripgrep 的三种部署模式

Claude Code 通过 utils/ripgrep.ts（680行）的 getRipgrepConfig() 函数，实现了 ripgrep 在不同环境下的无缝部署：

▶ ripgrep 部署模式选择逻辑 ripgrep.ts:31-65
// 三种模式的优先级：用户请求 system > embedded > builtin
const getRipgrepConfig = memoize((): RipgrepConfig => {
  // 模式 1: system — 用户系统安装的 rg
  if (userWantsSystemRipgrep) {
    const { cmd: systemPath } = findExecutable('rg', [])
    if (systemPath !== 'rg') {
      return { mode: 'system', command: 'rg', args: [] }
    }
  }
  // 模式 2: embedded — 静态编译进 bun 二进制
  // 通过 argv0='rg' 让同一个 bun 进程以 ripgrep 身份运行
  if (isInBundledMode()) {
    return {
      mode: 'embedded',
      command: process.execPath, // bun 自身
      args: ['--no-config'],
      argv0: 'rg', // 魔法：用 argv[0] 切换执行模式
    }
  }
  // 模式 3: builtin — vendor 目录中的预编译二进制
  const command = path.resolve(rgRoot,
    `${process.arch}-${process.platform}`, 'rg')
  return { mode: 'builtin', command, args: [] }
})

FileIndex：文件名补全的 Fuzzy Matcher

native-ts/file-index/index.ts（371行）实现文件名模糊匹配（用于 @ 提及补全），是从 Rust 的 nucleo 移植的纯 TypeScript 版本。这是字符级模糊匹配，不是语义搜索：

▶ FileIndex 核心评分算法 file-index/index.ts:208-271

  let pos = haystack.indexOf(needleChars[0]!)
  if (pos === -1) continue
  // 评分：SCORE_MATCH(16) + BONUS_BOUNDARY(8) + BONUS_CAMEL(6) + BONUS_CONSECUTIVE(4)
  // 对含 "test" 的路径施加 1.05x 惩罚，让非测试文件排名更高

// 注意：完全没有 embedding、向量、语义——纯字符匹配
🔑 设计哲学——"AI 做决策，传统工具做执行"：不是让 AI 做模糊匹配（AI 弱项是精确搜索），而是让 AI 决定搜什么参数（grep pattern），让 ripgrep 做精确搜索。这是 Claude Code 区别于 Cursor/Cody（依赖 RAG）的核心架构理念。

Prompt Cache 分层：DYNAMIC_BOUNDARY

System Prompt 被一个标记 SYSTEM_PROMPT_DYNAMIC_BOUNDARY 分为两段：

段缓存策略内容变化频率
静态段（BOUNDARY 之前）cacheScope: 'global'行为准则、工具使用指导、风格要求几乎不变（跨用户复用）
动态段（BOUNDARY 之后）每次重新计算记忆文件、环境信息、MCP 指令每次会话不同
💡 缓存分离的意义：Anthropic 的 Prompt Cache 按前缀匹配——如果 System Prompt 开头不变，后续即使变了也能复用缓存。通过 DYNAMIC_BOUNDARY 把"不变的"放前面、"变的"放后面，最大化缓存命中率，节省 API 成本。

16.8 关键设计决策总结

决策选择为什么不选替代方案
代码搜索引擎ripgrepRAG 需要基础设施、有索引过期问题、语义匹配不如精确匹配适合代码搜索
文件名补全nucleo fuzzy match字符级匹配比语义搜索更符合程序员的 tab 补全习惯
记忆召回Sonnet sideQuery比 embedding 更灵活（能理解项目上下文），200 个文件内 LLM 选择足够高效
记忆存储纯文本 .md 文件比数据库更透明、可 Git 管理、用户可直接编辑
上下文缓存lodash memoize比 Redis 更轻量，CLI 工具不需要跨进程缓存
规则条件化frontmatter paths glob比编程式配置更直觉，直接写在 .md 文件头部
Prompt CacheDYNAMIC_BOUNDARY 分段前缀不变则最大化 Anthropic Prompt Cache 命中率
🔑 一句话总结：Claude Code 的上下文管理哲学是"零基础设施 + AI 做决策 + 传统工具做执行"——不需要 Docker、不需要向量数据库、不需要预建索引。所有搜索都是实时文件系统扫描，所有记忆都是纯文本文件，所有"智能"选择都交给 LLM sideQuery。这符合 CLI 工具"安装即用"的产品定位。
