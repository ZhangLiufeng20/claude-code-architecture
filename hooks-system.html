🪝 React Hooks 层 19

🔬 方法论：本章对 src/hooks/（104 文件 / 16,476 行）进行源码级深度分析。 核心文件：useTypeahead.tsx（1384行）· useVoice.ts（1144行）· useInboxPoller.ts（969行）· fileSuggestions.ts（811行）· useReplBridge.tsx（722行）· useVirtualScroll.ts（721行）· useTextInput.ts（529行）· useRemoteSession.ts（605行）。

hooks/ 不是简单的自定义 Hook 集合——它是 Claude Code 的 React 状态管理层。 这 104 个文件围绕 4 个核心职责展开：输入管线（将键盘事件转换为结构化命令）、虚拟化渲染（在 1000+ 消息会话中维持 O(1) 内存占用）、语音交互（hold-to-talk + WebSocket STT）、协作通信（多设备/多代理消息路由）。

19.1 输入管线（Typeahead + TextInput）

useTypeahead（1384行）和 useTextInput（529行）组成了 Claude Code 的输入处理核心。用户每次按键都经过以下管线：

7 种建议管线优先级

当用户输入时，updateSuggestions 按以下优先级分发：

  1. mid-input slash command ghost text（prompt 模式，如 help me /com → /compact 幽灵补全）
  2. bash history ghost text（bash 模式，异步 getShellHistoryCompletion）
  3. @ 团队成员/Agent（从 AppState.teamContext.teammates + agentNameRegistry 实时读取）
  4. # Slack 频道（通过 MCP 服务器调用，150ms debounce）
  5. @ 文件路径（debouncedFetchFileSuggestions，50ms debounce）
  6. / 斜杠命令（同步 generateCommandSuggestions）
  7. 参数提示（generateProgressiveArgumentHint）

Emacs Readline 键绑定

useTextInput 实现了完整的 Emacs readline 兼容层，包括 Kill Ring 机制：

类别键操作

Ctrl-E行尾
Ctrl-B/F左/右移一字符
Meta-B/F前/后跳一词

19.2 虚拟滚动引擎（useVirtualScroll）

useVirtualScroll（721行）解决了一个核心性能问题：1000+ 消息会话中的 ~250MB RSS 内存增长。每个 MessageRow 约占 250KB（React fiber + Yoga 节点 + Ink 屏幕缓冲），全部挂载会导致内存失控。

核心算法参数

常量值设计理由
DEFAULT_ESTIMATE3 行故意偏低——高估导致空白底部，低估只会多挂载几项（被 overscan 吸收）

useDeferredValue 时间切片

范围 增长（新挂载需要 marked.lexer + formatToken ≈ 3ms/项）通过 useDeferredValue 延迟到非阻塞后台渲染：

const dStart = useDeferredValue(start);
const dEnd = useDeferredValue(end);
// 仅延迟增长方向，收缩（unmount = 删除 fiber）免费
let effStart = start < dStart ? dStart : start;
let effEnd = end > dEnd ? dEnd : end;
// sticky 时跳过延迟——scrollToBottom 需要尾部立即挂载
if (effStart > effEnd || isSticky) { effStart = start; effEnd = end; }

Resize 冻结策略

终端列数变化时，高度缓存全部失效。直接清空会导致 PESSIMISTIC_HEIGHT=1 回退最坏情况（挂载 ~190 项 → 600ms 阻塞）：

  1. 按比例缩放：heightCache.set(k, Math.max(1, Math.round(h * ratio))) — 保持估计「大致正确」
  2. 冻结 2 帧：freezeRendersRef.current = 2 — 第 1 帧跳过测量（Yoga 仍是旧宽度），第 2 帧 useLayoutEffect 读取正确高度
  3. Clamp 防抖：setClampBounds(clampMin, clampMax) 告诉 render-node-to-output 将 scrollTop 限制在已挂载范围内

18 种语言 + BCP-47 规范化

📄 normalizeLanguageForSTT 语言降级链 
export function normalizeLanguageForSTT(language: string | undefined): {
  code: string; fellBackFrom?: string
} {
  if (!language) return { code: 'en' };       // 1. 空值 → 默认英语
  const lower = language.toLowerCase().trim();
  if (SUPPORTED_LANGUAGE_CODES.has(lower))     // 2. 直接 BCP-47 码
    return { code: lower };
  const fromName = LANGUAGE_NAME_TO_CODE[lower];// 3. 语言名称映射（含原生名：日本語/한국어/हिन्दी）
  if (fromName) return { code: fromName };
  const base = lower.split('-')[0];            // 4. 基础子标签（pt-BR → pt）
  if (base && SUPPORTED_LANGUAGE_CODES.has(base))
    return { code: base };
  return { code: 'en', fellBackFrom: language }; // 5. 终极降级 → en + 警告标记
}

Silent-Drop 自动重放

~1% 的 voice_stream 会话遭遇「粘性 CE Pod」bug（PR#287008）——服务端接受音频但返回零转录。检测条件：

  1. finalizeSource === 'no_data_timeout'（finalize 超时而非正常结束）
  2. hadAudioSignal === true（麦克风有真实信号，非静音）
  3. wsConnected === true（WebSocket 已连接，音频到达了服务端）
  4. accumulatedRef.current.trim() === ''（零转录输出）

满足全部条件时，自动将 fullAudioRef 中缓存的完整音频（~2MB 上限）在 250ms 退避后通过新连接重放，以 32KB 切片发送。

RMS 音频电平计算

export function computeLevel(chunk: Buffer): number {
  const samples = chunk.length >> 1;  // 16-bit = 2 bytes/sample
  let sumSq = 0;
  for (let i = 0; i < chunk.length - 1; i += 2) {
    const sample = ((chunk[i]! | (chunk[i + 1]! << 8)) << 16) >> 16; // LE signed
    sumSq += sample * sample;
  }
  const rms = Math.sqrt(sumSq / samples);
  return Math.sqrt(Math.min(rms / 2000, 1)); // sqrt curve 扩展安静电平视觉范围
}

19.4 协作与通信

三个核心 Hook 组成了 Claude Code 的多设备/多代理通信骨干：

useInboxPoller — 10 种消息类型路由

useInboxPoller（969行）每 1s 轮询 readUnreadMessages()，将收到的消息分类路由到 10 个处理管线：

useReplBridge — claude.ai ↔ CLI 桥

useReplBridge（722行）建立与 claude.ai 的 WebSocket 双向通道，核心安全设计：

  • 编译时特性门控：feature('BRIDGE_MODE')（bun:bundle）— 外部构建中完全 tree-shaken
  • 3 连续失败熔断：MAX_CONSECUTIVE_INIT_FAILURES=3 → 永久禁用本次会话的桥重试（Datadog 显示单个客户日产 2,879 × 401）
  • 10s 自动禁用：BRIDGE_FAILURE_DISMISS_MS=10_000 → 失败后自动清除 replBridgeEnabled
  • Perpetual 模式：KAIROS 助手模式下跳过 teardown，保持跨 CLI 重启的会话连续性
  • 隐私保护：REPL-bridge 发送 system/init 时 tools=[]、mcpClients=[]、plugins=[] — 防止 MCP 工具名/路径泄露到 Spanner

useRemoteSession — SDK 消息转换

useRemoteSession（605行）管理远程 CCR 会话，关键机制：

  • BoundedUUIDSet(50) 去重：防止 POST 用户消息的 WebSocket 回显导致双重显示（相同 UUID 可回显 2 次——服务器广播 + Worker 回写）
  • 60s/180s 卡死检测：正常 60s 超时 + compaction 期间 180s 扩展超时 → 超时后自动 reconnect + 警告消息
  • Subagent 计数：通过 task_started/task_notification SDK 事件跟踪远程子代理运行数，驱动 UI「N in background」计数器
  • Compaction 状态追踪：isCompactingRef 跟踪 status='compacting' → 抑制虚假「无响应」警告

useQueueProcessor — 统一命令队列

useQueueProcessor（69行）是最精简但最关键的调度器——通过 useSyncExternalStore 订阅模块级命令队列，当满足以下 3 个条件时触发处理：

  1. !isQueryActive（无查询正在运行 — 由 QueryGuard 驱动）
  2. queueSnapshot.length > 0（队列有待处理项）
  3. !hasActiveLocalJsxUI（无本地 JSX UI 阻塞输入）

优先级排序由 dequeue() 自动处理：'now' > 'next'（用户输入） > 'later'（任务通知）。

🔑 一句话总结：hooks/ 是 Claude Code 的 React 状态管理层。输入管线通过 Unicode-aware token 解析和 7 种建议管线实现智能补全；虚拟滚动引擎以 O(log n) 二分查找 + 40 行量化将 1000+ 消息会话的内存维持在可控范围；语音子系统提供 hold-to-talk + 18 语言 STT + silent-drop 自动重放；协作通信通过 InboxPoller/ReplBridge/RemoteSession 三层架构实现多设备多代理消息路由。这 104 个 Hook 共同驱动了 Claude Code 从键盘事件到 UI 渲染的整个 React 生命周期。
